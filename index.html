Overview

OKLCH Gamut Mapping

When computing colors mathematically in perceptually uniform color spaces, results routinely fall outside what any display can actually show. This page covers why that happens, how the Chroma Reduction method solves it elegantly, and how it compares against three alternatives.

What is a Display Gamut?

Every display technology — LCD, OLED, CRT — can only reproduce colors within a bounded region of all perceivable color. The most common target is sRGB, defined as a triangle in the CIE xy chromaticity diagram. Colors "inside the triangle" are displayable; everything else is out-of-gamut.

When you work in a perceptually uniform color space like OKLab or OKLCH and then try to convert a vibrant color to sRGB, you'll frequently get values like R = 1.42, G = 0.61, B = −0.08. Those values are mathematically valid in OKLab but physically impossible to render. You must bring them in-gamut before display.

The Naive Approach: RGB Clamping

The simplest fix is to just clamp: R = min(1, max(0, R)). But clamping R = 1.42 → 1.0 and B = −0.08 → 0.0 changes the ratio between channels. Ratios encode hue. The color you get is a different color — wrong lightness, wrong hue. You can see exactly this distortion in the "Clip" panel of the comparison grid below.

A Better Strategy

The insight behind chroma reduction is this: OKLCH represents colors as a lightness L, a hue angle H, and a chroma C (distance from the neutral grey axis). For any given L and H, there is a maximum chroma Cmax that fits within sRGB. If a color exceeds that, we simply reduce C until it doesn't — preserving both lightness and hue throughout.

Foundations

OKLab & OKLCH

OKLab (Björn Ottosson, 2020) is a perceptually uniform color space with three axes:

  • L — lightness, from 0 (black) to 1 (white)
  • a — green (negative) to red (positive)
  • b — blue (negative) to yellow (positive)

"Perceptually uniform" means equal Euclidean distances in (L, a, b) correspond to equal perceived color differences. This is why OKLab is far better than RGB or even HSL for color manipulation — operations like blending, darkening, and saturation adjustments produce predictably correct results.

The OKLab ab-plane at a fixed lightness L. Chroma C is the radial distance from the origin; hue H is the angle. The dashed circle represents a constant-chroma contour.

Cross-section of the sRGB gamut at a fixed hue H. The blue curve is the gamut boundary C_max(L). An out-of-gamut point (red) is mapped horizontally to the boundary (green), preserving L.

From OKLab to OKLCH

OKLCH is simply OKLab expressed in polar coordinates around the lightness axis. Given (L, a, b):

C = √(a² + b²) // chroma: distance from neutral axis H = atan2(b, a) × (180/π) // hue: angle in degrees [0, 360)

This decomposition is key: because C and H are independent, we can manipulate them separately. Reducing C while holding L and H constant means moving radially inward along the same hue ray — the color gets less saturated but doesn't shift in hue or lightness.

Algorithm

How Chroma Reduction Works

The algorithm rests on a single geometric fact: for any fixed lightness L and hue H, the set of in-gamut chromas forms a contiguous interval [0, C_max(L, H)]. There are no "holes" — if C = 0.15 and C = 0.25 are both in-gamut, every value between them is too.

This means we can binary-search for C_max, and then map any out-of-gamut color to its boundary equivalent in one clean operation:

// The complete gamut mapping pipeline ① Test if already in gamut linear_rgb = oklabToLinearSRGB(L, a, b) if all(0 ≤ r,g,b ≤ 1): return applyGamma(linear_rgb) // fast path, no search needed ② Extract chroma and hue C = √(a² + b²) H = atan2(b, a) · (180/π) ③ Binary search for maximum in-gamut chroma C_max = maxChroma(L, H) ④ Clamp chroma and convert return oklchToSRGB(L, min(C, C_max), H)
Why This Preserves Hue

The hue angle H is never touched. We only modify C. In the ab-plane, this means we're moving straight toward the origin along the same ray. The direction of that ray is what defines hue — so hue is perfectly preserved by construction.

The Fast Path

Most colors in a typical image are already in gamut. The algorithm checks this first, using a single matrix multiplication. If in-gamut, it returns immediately — no binary search needed. This makes the common case very fast.

Mathematics

The Conversion Math

OKLab → Linear sRGB

The conversion from OKLab to linear (pre-gamma) sRGB involves two steps. First, reconstruct intermediate values l', m', s' (representing LMS cone responses in a modified space):

l' = L + 0.3963377774·a + 0.2158037573·b m' = L − 0.1055613458·a − 0.0638541728·b s' = L − 0.0894841775·a − 1.2914855480·b l = l'³ m = m'³ s = s'³ // cube — undoes the cube-root in OKLab's definition

The cube-root step is central to OKLab's design — it's what gives the space its perceptual uniformity. The cube root compresses large values and expands small ones, mimicking how human vision responds to light intensity. The cube here undoes it, restoring physical LMS values.

Then apply the LMS → linear sRGB matrix (derived from the sRGB primaries via D65 white point):

R_lin = 4.0767416621·l − 3.3077115913·m + 0.2309699292·s G_lin = −1.2684380046·l + 2.6097574011·m − 0.3413193965·s B_lin = −0.0041960863·l − 0.7034186147·m + 1.7076147010·s

In-Gamut Test

The sRGB gamut in linear space is simply the unit cube — each channel must be in [0, 1]. We check this before doing any search:

inSRGB(R, G, B) = (R ≥ 0) ∧ (R ≤ 1) ∧ (G ≥ 0) ∧ (G ≤ 1) ∧ (B ≥ 0) ∧ (B ≤ 1)

Gamma Correction

Once we have linear sRGB values in [0, 1], we apply the standard sRGB transfer function (gamma) to get display-ready values:

f(c) = // IEC 61966-2-1 sRGB gamma if c ≤ 0.0031308: 12.92 · c // linear segment (shadow region) else: 1.055 · c^(1/2.4) − 0.055 // power curve

This piecewise function is applied to R, G, and B independently. The linear segment at the bottom avoids a kink at zero (a pure power curve would have an infinite slope at 0).

Algorithm Detail

Binary Search for Cmax

The function maxChroma(L, H) finds the largest C for which oklchToLinearSRGB(L, C, H) produces all channels in [0, 1]. It works in two phases.

Phase 1: Find an Upper Bound

We don't know how large C_max is, so we start at hi = 0.01 and double it until we go out of gamut (or reach 0.5, which is well beyond any sRGB-reachable chroma). This geometric search finds a bracket in O(log C_max) steps — usually 3–6 doublings.

hi = 0.01 while hi < 0.5 and inSRGB(oklchToLinearSRGB(L, hi, H)): hi = hi × 2 // bracket doubles until out of gamut // Now: lo = 0 (safe), hi = first out-of-gamut value found

Phase 2: Binary Search

With a bracket [lo, hi] where lo is in-gamut and hi is out-of-gamut, we binary-search for 25 iterations. Each iteration halves the interval. After 25 steps from an initial bracket of ~0.5, we have precision of 0.5 / 2²⁵ ≈ 1.5 × 10⁻⁸ — far beyond any practical need.

Code

Chroma Reduction — Annotated Code

The entire chroma reduction method is implemented in pure JavaScript with no dependencies. Here are all six functions explained in order.

1. Gamma Correction

Converts a single linear-light channel to sRGB display encoding.

linearToGamma(c) pure math · no alloc 
function linearToGamma(c) {
  // IEC 61966-2-1 sRGB transfer function.
  // Linear segment below the threshold prevents
  // an infinite derivative at c = 0.
  return c <= 0.0031308
    ? 12.92 * c                         // linear region
    : 1.055 * Math.pow(c, 1 / 2.4) - 0.055; // gamma curve
}

2. OKLab → Linear sRGB

The core conversion. Two matrix multiplications with a cube-root inversion between them.

oklabToLinearSRGB(L, a, b) 27 multiplications · returns [R, G, B] 
function oklabToLinearSRGB(L, a, b) {
  // Stage 1: multiply by the OKLab→LMS matrix.
  // These coefficients encode how the (L, a, b) axes
  // project onto each cone-response channel.
  const l_ = L + 0.3963377774 * a + 0.2158037573 * b;
  const m_ = L - 0.1055613458 * a - 0.0638541728 * b;
  const s_ = L - 0.0894841775 * a - 1.2914855480 * b;

  // Stage 2: undo OKLab's cube-root compression.
  // OKLab stores l^(1/3), m^(1/3), s^(1/3) — cube
  // these back to recover linear LMS cone responses.
  const l = l_ ** 3;
  const m = m_ ** 3;
  const s = s_ ** 3;

  // Stage 3: multiply by the LMS→linear sRGB matrix.
  // These coefficients come from the sRGB primaries
  // under D65 illuminant.
  return [
     4.0767416621 * l - 3.3077115913 * m + 0.2309699292 * s,  // R
    -1.2684380046 * l + 2.6097574011 * m - 0.3413193965 * s,  // G
    -0.0041960863 * l - 0.7034186147 * m + 1.7076147010 * s,  // B
  ];
}

3. In-Gamut Test

A simple unit-cube check. Must be done in linear (pre-gamma) space because that's where sRGB's gamut boundaries are straight planes.

function inSRGBLinear(rgb) {
  // sRGB gamut = unit cube [0,1]³ in linear space.
  // A value of 1.001 is just as out-of-gamut as 2.0 —
  // both are impossible to display.
  return rgb.every(v => v >= 0 && v <= 1);
}

4. OKLCH → Linear sRGB

Used exclusively inside the binary search. Converts a chroma value at a given hue back to (a, b), then to linear RGB.

function oklchToLinearSRGB(L, C, H) {
  // Convert polar (C, H) → Cartesian (a, b).
  // H is in degrees; convert to radians first.
  const h = H * Math.PI / 180;
  const a = C * Math.cos(h);
  const b = C * Math.sin(h);
  return oklabToLinearSRGB(L, a, b);
}

5. Maximum Chroma Search

The heart of the algorithm. Finds the gamut boundary at a given lightness and hue via exponential expansion followed by binary search.

maxChroma(L, H) ~6 bracket iters + 25 binary search iters · returns float 
function maxChroma(L, H) {
  // Phase 1: Exponential search to bracket C_max.
  // Start tiny (0.01) and double until out of gamut.
  // This finds an upper bound in O(log C_max) steps
  // without needing to know C_max ahead of time.
  let lo = 0, hi = 0.01;
  while (hi < 0.5 && inSRGBLinear(oklchToLinearSRGB(L, hi, H))) {
    hi *= 2;
  }
  // Invariant: lo is in-gamut, hi is out-of-gamut.

  // Phase 2: Binary search within [lo, hi].
  // 25 iterations → precision ~= (hi - lo) / 2^25 ≈ 1.5e-8.
  // That's way beyond floating-point noise in color values.
  for (let i = 0; i < 25; i++) {
    const mid = (lo + hi) / 2;
    if (inSRGBLinear(oklchToLinearSRGB(L, mid, H))) {
      lo = mid;  // mid is in-gamut → raise lower bound
    } else {
      hi = mid;  // mid is out-of-gamut → lower upper bound
    }
  }

  // lo ≈ hi ≈ C_max after 25 iterations.
  // Return lo (the last known in-gamut chroma).
  return lo;
}

6. The Main Map Function

Combines everything. Note the fast path: colors already in gamut skip all the search logic entirely.

function pickerMap(L, a, b) {
  // Edge cases: L = 0 and L = 1 are pure black/white.
  // No conversion needed — they map to {0,0,0}/{1,1,1}.
  if (L <= 0) return [0, 0, 0];
  if (L >= 1) return [1, 1, 1];

  // Fast path: check if already in sRGB gamut.
  // Most colors in typical usage will be in-gamut,
  // so this path is taken frequently.
  const rgb = oklabToLinearSRGB(L, a, b);
  if (inSRGBLinear(rgb)) {
    return rgb.map(linearToGamma);  // just apply gamma, done
  }

  // Slow path: the color is out of gamut.
  // Extract chroma and hue from (a, b).
  const C = Math.sqrt(a * a + b * b);
  let H = Math.atan2(b, a) * 180 / Math.PI;
  if (H < 0) H += 360;  // normalize to [0, 360)

  // Binary-search for the maximum in-gamut chroma
  // at this lightness and hue, then clamp to it.
  const C_mapped = Math.min(C, maxChroma(L, H));
  return oklchToLinearSRGB(L, C_mapped, H).map(linearToGamma);
}

Clip (~4 µs)

The absolute minimum: one matrix multiplication (OKLab → sRGB), then three Math.max / Math.min calls. Zero branching, zero allocation. It's fast precisely because it doesn't think — it just truncates. The price is visual correctness.

Chroma Reduction (~20–40 µs)

Pure arithmetic in hot, allocation-free loops. The binary search runs ~30 iterations total (≈5 bracket doublings + 25 bisections), each performing a single oklabToLinearSRGB call — 27 multiplications and 9 additions. No objects are created, no garbage is collected. V8 can JIT this aggressively. Colors already in gamut hit the fast path and cost only one oklabToLinearSRGB + 3 comparisons.

OKLCH + ΔEOK (~75 µs)

Structurally similar to Chroma Reduction, but routed through color.js — a full-featured library that represents colors as JavaScript objects. Every to() call, every clone(), every inGamut() creates new objects and walks prototype chains. There's also an additional ΔEOK check: before the binary search, it checks whether clipping would produce a perceptually close result (within the JND threshold of 0.02), and if so, skips the search. This is a useful quality-of-life optimization but adds overhead on the common path.

CIE LCH + ΔE2000 (~230 µs)

The most expensive method for two reasons. First, it operates in CIE LCH (older, less perceptually uniform than OKLCH), which requires an intermediate conversion through CIE XYZ and CIE Lab — more matrix multiplications and a more complex cube-root function. Second, ΔE2000 is an extremely involved formula: it includes hue rotation corrections, chroma weighting factors, and trigonometric compensation terms designed to fix known perceptual non-uniformities in CIE Lab. Computing ΔE2000 takes roughly 60–80 floating-point operations per call, and it's called at every binary search iteration. That's the dominant cost.

Code · Method 2

Clip — Channel Clamping

The simplest method: convert to sRGB (via color.js for consistency with the other library methods), then clamp each channel to its valid range. 

function clip(color, destination) {
  // Convert from OKLab to the destination space (sRGB).
  // This may produce channel values outside [0, 1].
  let destColor = to(color, destination);

  // Get the valid range for each channel in the destination space.
  // For sRGB: R ∈ [0, 1], G ∈ [0, 1], B ∈ [0, 1].
  const destSpace = Color.Space.get(destination);
  const bounds = Object.values(destSpace.coords).map(c => c.range || []);

  // Clamp each channel independently.
  // This is the source of hue distortion: R, G, B encode
  // color as a combination of primaries. Clamping R=1.4 to 1.0
  // while leaving G=0.5 unchanged alters their ratio, which
  // shifts the perceived hue of the result.
  destColor.coords = destColor.coords.map((c, i) => {
    const [min, max] = bounds[i];
    if (min !== undefined) c = Math.max(min, c);
    if (max !== undefined) c = Math.min(c, max);
    return c;
  });

  return destColor;
}

// Example: oklch(0.65, 0.27, 32°) → clip
// R_lin = 1.12  → clamped to 1.0  (lost 0.12)
// G_lin = 0.06  → unchanged
// B_lin = 0.04  → unchanged
// The lost R shifts the apparent hue toward cyan/green.
Code · Method 3

OKLCH + ΔEOK — The CSS Color 4 Algorithm

The proposed CSS Color 4 gamut mapping algorithm. It works like Chroma Reduction, but with an important refinement: before the full binary search, it checks whether a simple clip would be perceptually close enough (within the JND — Just Noticeable Difference) to the correct answer. If so, it takes the shortcut.

What is ΔEOK?

ΔEOK is a perceptual difference metric in OKLab space. It's simply the Euclidean distance between two colors: √(ΔL² + Δa² + Δb²). A value below 0.02 (the JND threshold) means two colors are likely indistinguishable to the human eye. The CSS GMA uses this to avoid expensive binary searching when the clip result is already "good enough."

const JND = 0.02;   // Just Noticeable Difference threshold
const ε   = 0.0001; // convergence tolerance for binary search

function CSSgamutMap(origin, destination) {
  // Work in OKLCH — it has the best perceptual properties
  // for this kind of chroma manipulation.
  const origin_OKLCH = to(origin, 'oklch');
  const L = origin_OKLCH.coords[0];

  // Achromatic edge cases: black and white.
  if (L >= 1) return { space: destination, coords: [1, 1, 1], alpha: origin.alpha };
  if (L <= 0) return { space: destination, coords: [0, 0, 0], alpha: origin.alpha };

  // Fast path: already in gamut.
  if (inGamut(origin, destination)) return to(origin, destination);

  // Pre-check: would clipping produce a nearly-correct result?
  // This avoids the binary search for colors close to the boundary.
  let current = clone(origin_OKLCH);
  let clipped  = clip(current, destination);
  if (deltaEOK(clipped, current) < JND) return clipped;
  // ↑ If the clipped result is within JND of the original
  //   (measured in OKLab), the error is imperceptible. Skip search.

  // Full binary search: reduce chroma until in-gamut.
  let min = 0, max = origin_OKLCH.coords[1];
  let min_inGamut = true;

  while (max - min > ε) {
    const chroma = (min + max) / 2;
    current.coords[1] = chroma;  // update chroma in-place

    if (min_inGamut && inGamut(current, destination)) {
      min = chroma;  // raise floor
    } else {
      clipped = clip(current, destination);
      const E = deltaEOK(clipped, current);
      if (E < JND) {
        // Clipping at this chroma is perceptually close.
        if (JND - E < ε) return clipped;  // close enough, stop
        min_inGamut = false;
        min = chroma;
      } else {
        max = chroma;  // still too far out, reduce more
      }
    }
  }

  return current;
}

The key difference from the vanilla Chroma Reduction: the CSS GMA's binary search also checks deltaEOK(clipped, current) at each step. This allows it to converge on a point slightly outside the strict gamut boundary if the clip error there is below JND — producing a slightly "looser" result that can look slightly more vibrant in edge cases. In practice, the visual difference is negligible.

CIE LCH + ΔE2000 — The color.js Default

The oldest of the four methods. It performs the same chroma binary search, but in CIE LCH space using ΔE2000 as the perceptual metric. This was the standard approach before OKLCH existed — historically important, but now largely superseded.

CIE LCH vs OKLCH

CIE LCH is derived from CIE Lab (1976), which has known non-uniformities — blues and purples in particular are perceptually distorted. OKLCH corrects these by using a more carefully fitted perceptual model. OKLCH's ΔEOK is also simpler to compute (just Euclidean distance) vs the complex correction terms in ΔE2000.

function render_cjs(L = 0.5) {
  // color.js's toGamut() defaults to CIE LCH with ΔE2000.
  // It performs a binary search on CIE chroma until
  // the color maps into the destination gamut.

  for (let a = -0.4; a <= 0.4; a += 0.016) {
    for (let b = -0.4; b <= 0.4; b += 0.016) {

      // 1. Start in OKLab space (our input representation).
      const OKLab = { space: 'oklab', coords: [L, a, b] };

      // 2. Convert to sRGB via color.js.
      //    This will produce out-of-range values if the
      //    OKLab color is outside the sRGB gamut.
      const color = to(OKLab, 'srgb');

      // 3. toGamut() applies the CIE LCH gamut mapping algorithm.
      //    Internally: convert to CIE LCH → binary search on C
      //    → check ΔE2000 at each step → clamp.
      //
      //    ΔE2000 cost breakdown:
      //      - Convert both colors to CIE Lab
      //      - Compute ΔL', ΔC', ΔH' with rotation corrections
      //      - Apply weighting functions SL, SC, SH
      //      - Apply RT (rotation term, depends on average hue)
      //    Total: ~70 FP operations per call vs ~3 for ΔE_OK.
      toGamut(color, 'srgb');

      // 4. Render the pixel.
      plot(ctx, color, a, b);
    }
  }
}

// - Legacy compatibility with existing color.js pipelines
// - ΔE2000 may handle certain hue-angle edge cases better
// - Some workflows require strict CIE Lab compliance
// For new work, OKLCH + ΔE_OK is preferred at 3× less cost.
